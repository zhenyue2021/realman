## 1. 执行摘要

### 1.1 背景

RealMan IoT 平台采用 Spring Boot 3.5.5 + Spring Cloud 微服务架构，通过 Nginx 负载均衡实现多节点集群部署。当前系统存在四个核心痛点：

- **无链路追踪能力**：一次请求跨 HTTP、Feign、MQTT、异步线程多个环节，无法关联全链路日志，问题定位依赖经验和猜测
- **日志散落各节点**：集群模式下每个 Pod 独立写文件，排查问题需逐台 SSH 登录，时效性极差
- **无结构化日志**：纯文本格式输出，无法按字段检索、过滤、聚合，无法支撑自动化告警
- **MQTT 消息链路断裂**：设备上报/平台下发指令的 MQTT 消息经 EMQX 共享订阅分发后，跨 Pod 处理时链路彻底丢失

### 1.2 方案目标

建立一套 **轻量级、低侵入、可演进** 的分布式可观测性体系，达成以下目标：

| 目标 | 具体描述 | 验收标准 |
|------|---------|---------|
| 全链路 TraceId 贯穿 | HTTP → Feign → MQTT → Async → XXL-Job 全场景覆盖 | 任意请求可通过单个 traceId 查看所有关联日志 |
| 集群日志统一聚合 | 所有节点日志集中存储，按 traceId / 设备 / 节点 / 时间多维查询 | Grafana 单一界面完成全部日志查询 |
| 三板斧联动 | 日志（Loki）+ 链路拓扑（Zipkin）+ 指标（Prometheus）统一可视化 | 从日志点击 traceId 可直接跳转链路拓扑图 |
| 业务零侵入 | 链路追踪对现有业务代码改动 ≤ 5 处 | 无需修改 Controller / Service 层代码 |

### 1.3 核心收益（对比 ELK 方案）

| 指标 | ELK 方案 | 本方案（Loki+Zipkin+Grafana） | 节省 |
|------|---------|------|------|
| 内存占用 | ≥ 6 GB | ≈ 768 MB | **87%** |
| CPU 占用 | 4 Core | 1 Core | **75%** |
| 存储空间（30天） | ~500 GB | ~50 GB | **90%** |
| 部署复杂度 | 高（ES 调优 + Logstash 管道） | 低（Docker 一键启动） | — |
| 部署周期 | 7-10 人天 | 4 人天 | **60%** |

### 1.4 范围与约束

| 维度 | 说明 |
|------|------|
| 技术栈 | Spring Boot 3.5.5 / Spring Cloud / JDK 17+ |
| 服务节点 | realman-cloud-gateway（WebFlux）、realman-boot-iot（Servlet）、realman-boot-system（Servlet） |
| 通信协议 | HTTP/REST、OpenFeign、MQTT 5.0（EMQX 5.x） |
| 已有监控 | Micrometer + Prometheus + Grafana（已部署） |
| 部署方式 | Docker Compose 集群，Nginx 负载均衡 |
| 核心约束 | Spring Boot 3.x 已废弃 Sleuth，必须使用 Micrometer Tracing |

## 2. 需求分析

### 2.1 功能需求

#### FR-T：链路追踪

| 编号 | 需求 | 详细描述 |
|------|------|---------|
| FR-T1 | HTTP 请求链路追踪 | 从 Gateway 入口生成 TraceId，自动为每个服务调用生成 Span，记录开始/结束时间、Tags、Events |
| FR-T2 | MQTT 消息链路追踪 | 发送时将 TraceId 写入 MQTT 5.0 User Properties，接收时恢复到 MDC；设备主动上报时生成新 TraceId |
| FR-T3 | 异步任务链路追踪 | @Async 方法继承父线程 TraceId；XXL-Job 定时任务生成独立 TraceId（格式：`job-{jobId}-{timestamp}`） |
| FR-T4 | Feign 调用透传 | 跨服务调用时自动透传 B3 Header 及自定义 X-Trace-Id |
| FR-T5 | 链路可视化 | 支持 Zipkin UI / Grafana 查看完整链路拓扑；支持日志与链路的联动跳转 |

#### FR-L：日志管理

| 编号 | 需求 | 详细描述 |
|------|------|---------|
| FR-L1 | 结构化日志 | 所有日志 JSON 格式输出，包含 traceId、spanId、service、instance、source 等上下文字段 |
| FR-L2 | 日志聚合 | 多服务多实例日志集中存储于 Loki，支持按服务/实例/级别/关键字/时间范围过滤 |
| FR-L3 | 日志保留 | 支持按时间保留（默认 30 天）；本地文件滚动备份（100MB 单文件，30 天保留） |
| FR-L4 | 日志告警 | 支持 ERROR 日志速率告警、OOM 关键字告警、服务离线告警 |
| FR-L5 | 日志脱敏 | 密码、Token、手机号、身份证、银行卡等敏感信息自动脱敏 |

### 2.2 非功能需求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 链路追踪延迟 | < 100 ms | Brave SDK 内存中采集，异步批量上报 Zipkin |
| 日志推送延迟 | < 500 ms | loki4j 异步批量推送（5 秒间隔 / 100 条批次） |
| 采样率 | 生产 10% / 测试 100% | 通过 `management.tracing.sampling.probability` 控制 |
| 日志丢失率 | < 0.1% | loki4j 失败重试 3 次（指数退避）+ 本地文件兜底 |
| 链路查询响应 | < 2 s | Loki 按标签索引 + Zipkin 内存存储 |
| 可用性 | 追踪系统故障不影响业务 | 所有采集组件异步运行，失败静默降级 |

### 2.3 安全需求

| 需求 | 实现方式 |
|------|---------|
| 传输加密 | Loki / Zipkin 内网通信；如需外网访问，配置 TLS |
| 敏感信息脱敏 | Logback 自定义 Layout 正则替换（详见 6.4 节） |
| 访问控制 | Grafana RBAC 角色控制日志查看权限 |

## 3. 技术选型与对比分析

### 3.1 候选方案对比

在设计过程中评审了三种方案：

| 维度 | 方案A: Redis+MySQL 自研 | 方案B: ELK Stack | 方案C: Loki+Grafana（✅ 推荐） |
|------|----------------------|-----------------|--------------------------|
| 链路追踪 SDK | Micrometer + OTel | Micrometer + Brave | Micrometer + Brave |
| 日志存储 | MySQL 分区表 | Elasticsearch | Loki |
| 日志采集 | 自定义 Redis Stream Appender | LogstashTcpSocketAppender | loki4j 直推 |
| 资源开销 | 中（Redis + MySQL） | **高（ES ≥ 2GB + Logstash 1GB）** | **低（Loki ~256MB）** |
| 可视化 | 自建 REST API（无 UI） | Kibana + Zipkin（两套系统） | Grafana 统一三板斧 |
| 运维复杂度 | 高（自维护消费者/分区表/清理） | 中（ES 分片/调优/扩容） | 低（开箱即用） |
| MQTT 透传方式 | User Properties | Payload 嵌入 `_traceCtx` | User Properties |
| Gateway WebFlux 处理 | ❌ 未覆盖 | ✅ 适配 | ✅ 适配 |
| 与已有 Grafana 生态融合 | 差（独立体系） | 中（需新增 Kibana） | **优（直接复用 Grafana）** |
| 全文检索能力 | 差（MySQL LIKE） | 强（ES 倒排索引） | 中（行过滤 + 正则） |

### 3.2 方案 A 淘汰理由

- 自建 `RedisStreamLogAppender` → `LogStreamConsumer` → `sys_trace_log` MySQL 表的链路，本质是重复造轮子
- 消费者可靠性（消息堆积、消费延迟、ACK 丢失）需要自行处理
- MySQL 分区表维护（定期创建分区、清理过期分区）增加运维负担
- 无可视化界面，只有 REST API，排查效率低

### 3.3 方案 B 淘汰理由

- Elasticsearch 单节点至少 2-4 GB 内存，加上 Logstash 约 1 GB，远超 IoT 项目基础设施预算
- Kibana 与已有 Grafana 形成两套可视化系统，运维分裂
- MQTT 透传采用 payload 嵌入 `_traceCtx` 字段，侵入业务数据结构，下游解析器全部要适配
- IoT 场景日志查询模式简单（按 traceId / 设备 / 时间），不需要 ES 的全文检索能力

### 3.4 最终选型决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 链路追踪 SDK | **Micrometer Tracing + Brave Bridge** | Spring Boot 3.x 官方方案（Sleuth 已废弃）；与已有 `micrometer-registry-prometheus` 自然协同 |
| 链路后端 | **Zipkin** | 轻量开箱即用，与 Brave Bridge 原生集成；后续可平滑升级到 Grafana Tempo |
| 日志存储 | **Grafana Loki** | 无需 ES 内存开销；仅索引标签而非全文，IoT 场景按 traceId/标签查询足够；与 Grafana 生态统一 |
| 日志采集 | **loki4j Logback Appender 直推** | 免 Promtail / Filebeat，减少部署组件；Java 进程内直推延迟更低 |
| 可视化 | **Grafana** | 统一展示日志（Loki）+ 链路（Zipkin）+ 指标（Prometheus），一个平台解决所有可观测性需求 |
| MQTT 传播 | **MQTT 5.0 User Properties** | 协议层透传，不污染业务 payload；MQTT 3.1.1 时 fallback 到 payload 嵌入 |

### 4.2 数据流向

#### 场景 1：HTTP 请求链路

```
外部请求 (无 traceId)
  → Gateway TraceIdGatewayFilter
    ├─ 生成 traceId = UUID
    ├─ 写入响应头 X-Trace-Id（前端可用）
    └─ 注入下游请求头 X-Trace-Id
  → IoT Service TraceIdMdcFilter
    ├─ 从 Header 读取 traceId
    ├─ 写入 MDC: {traceId, spanId, service, instance, source=http}
    └─ Logback 自动在每条日志中带上 traceId
  → Feign 调用 System Service
    ├─ Micrometer Tracing 自动注入 B3 Header
    ├─ FeignTraceInterceptor 额外透传 X-Trace-Id
    └─ System Service MDC Filter 读取 → 同一 traceId 贯穿
  → 请求返回
    └─ TraceIdMdcFilter finally 块清理 MDC（防止线程池污染）
```

#### 场景 2：MQTT 消息链路（跨 Pod 核心难点）

```
平台下发指令 (当前线程 MDC 已有 traceId=abc123)
  → TracingMqttPublisher
    ├─ 从 Micrometer Tracer 获取当前 traceId + spanId
    ├─ 写入 MQTT 5.0 User Properties: {X-Trace-Id=abc123, X-Span-Id=...}
    └─ 发送 MQTT 消息（payload 不做任何修改）
  → EMQX Broker 共享订阅分发
    └─ 消息路由到另一个 Pod
  → MqttMessageDispatcher (另一个 Pod)
    ├─ 从 User Properties 提取 X-Trace-Id
    ├─ 若存在 → 恢复 traceId 到 MDC（链路连续）
    ├─ 若不存在（设备主动上报）→ 生成 mqtt-{UUID} 格式新 traceId
    ├─ 写入 MDC: {traceId, source=mqtt, mqttTopic=...}
    └─ 后续所有 handler/service 日志自动带 traceId
  → Handler 处理完毕
    └─ finally 块清理 MDC
```

#### 场景 3：异步 / 定时任务

```
@Async 方法调用:
  主线程 MDC: {traceId=abc123}
  → AsyncMdcTaskDecorator
    ├─ 提交时快照主线程 MDC
    └─ 子线程执行前恢复 MDC → 执行完清理

XXL-Job 定时任务:
  → JobHandler 入口
    ├─ 生成 traceId = job-{jobId}-{timestamp}
    ├─ 写入 MDC
    └─ 任务执行期间所有日志关联同一 traceId
  → 任务完成
    └─ 清理 MDC
```

## 5. 链路追踪详细设计

### 5.1 术语定义

| 术语 | 定义 |
|------|------|
| TraceId | 链路追踪 ID，唯一标识一次完整请求链路，跨所有服务节点不变。格式为 32 位十六进制（W3C TraceContext 兼容）或 UUID |
| SpanId | 跨度 ID，标识链路中的单个操作单元，每个服务节点生成新 SpanId |
| MDC | Mapped Diagnostic Context，Logback 提供的线程级别日志上下文容器 |
| B3 Propagation | Zipkin 定义的链路上下文传播协议，通过 HTTP Header（X-B3-TraceId 等）透传 |
| User Properties | MQTT 5.0 协议新增的消息元数据机制，键值对格式，不影响 payload |

### 5.2 TraceId 传播矩阵

| 场景 | 入口 | 传播机制 | 存储 |
|------|------|---------|------|
| HTTP 请求 | X-Trace-Id Header | B3 Multi Header（Micrometer 自动） | MDC |
| Feign 调用 | MDC → Request Header | Micrometer 自动注入 B3 + FeignTraceInterceptor | MDC |
| MQTT 发送 | MDC | User Properties: X-Trace-Id, X-Span-Id | 消息元数据 |
| MQTT 接收 | User Properties | MqttMessageDispatcher 解析 → MDC | MDC |
| @Async | 父线程 MDC | AsyncMdcTaskDecorator 快照/恢复 | 子线程 MDC |
| XXL-Job | 无（链路起点） | Handler 入口生成 `job-{jobId}-{ts}` | MDC |
| RestTemplate | MDC | Micrometer 自动注入 B3 Header | MDC |

### 5.3 Span 类型定义

| Span 类型 | 操作名称示例 | 关键 Tags |
|-----------|------------|-----------|
| HTTP Server | `GET /api/device/list` | http.method, http.url, http.status_code |
| HTTP Client (Feign) | `POST /sys/api/checkAuth` | http.method, peer.service |
| Database | `SELECT iot_device` | db.type=mysql, db.statement |
| Cache | `GET device:cache:123` | cache.type=redis, cache.hit |
| MQTT Publish | `mqtt.publish.device/command` | messaging.system=mqtt, messaging.destination |
| MQTT Subscribe | `mqtt.subscribe.device/status` | messaging.system=mqtt, messaging.operation |
| Async | `async.processDeviceData` | component=async |
| Scheduled Job | `job.deviceSync` | job.id, job.name |

### 5.4 采样策略

```yaml
# application.yml
management:
  tracing:
    sampling:
      probability: ${TRACE_SAMPLING:1.0}  # 环境变量控制，默认 100%

# 环境差异化配置
# application-dev.yml  → TRACE_SAMPLING=1.0  (开发: 100%)
# application-prod.yml → TRACE_SAMPLING=0.1  (生产: 10%)
```

> **设计决策**：v2.0 方案提出了按操作名差异化采样率（如关键操作 100%，列表查询 5%）。当前版本暂不实现，预留为后续优化项。Micrometer Tracing 支持自定义 `Sampler` Bean，届时可平滑扩展。

## 6. 日志管理详细设计

### 6.1 结构化日志格式

所有服务统一采用 JSON 格式输出，通过 `logstash-logback-encoder` 实现。每条日志包含以下字段：

```json
{
  "@timestamp": "2026-04-10T10:23:45.123+08:00",
  "level": "INFO",
  "logger": "o.j.m.d.service.impl.IotDeviceServiceImpl",
  "thread": "http-nio-8080-exec-1",
  "message": "设备上线: deviceCode=robot001",
  "traceId": "abc123def456",
  "spanId": "7890abcd",
  "service": "realman-iot",
  "instance": "iot-node-2",
  "source": "http",
  "stack_trace": null
}
```

| 字段 | 类型 | 来源 | 说明 |
|------|------|------|------|
| @timestamp | String | Logback 自动 | ISO 8601 格式，毫秒精度 |
| level | String | Logback 自动 | DEBUG / INFO / WARN / ERROR |
| logger | String | Logback 自动 | 日志记录器（缩略类名） |
| thread | String | Logback 自动 | 线程名称 |
| message | String | 业务代码 | 日志正文 |

#### 7.4.2 Gateway TraceIdGatewayFilter

```java
// realman-cloud-gateway: filter/TraceIdGatewayFilter.java
@Component
@Order(-1)
public class TraceIdGatewayFilter implements GlobalFilter {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();
        String traceId = request.getHeaders().getFirst(TraceIdConst.HEADER_TRACE_ID);
        if (!StringUtils.hasText(traceId)) {
            traceId = UUID.randomUUID().toString().replace("-", "");
        }
        // ⚠️ WebFlux 响应式线程模型不能使用 MDC
        // Gateway 仅做 Header 透传，日志记录在下游 Servlet 服务
        exchange.getResponse().getHeaders().add(TraceIdConst.HEADER_TRACE_ID, traceId);
        String finalTraceId = traceId;
        return chain.filter(exchange.mutate()
            .request(r -> r.header(TraceIdConst.HEADER_TRACE_ID, finalTraceId))
            .build());
    }
}
```

#### 7.4.3 TraceIdMdcFilter（Servlet 服务核心）

```java
// org/jeecg/common/trace/TraceIdMdcFilter.java
@Component
@Order(-100)
public class TraceIdMdcFilter implements Filter {

    @Value("${spring.application.name:unknown}")
    private String serviceName;

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) throws IOException, ServletException {
        HttpServletRequest req = (HttpServletRequest) request;
        HttpServletResponse res = (HttpServletResponse) response;

        // 1. 优先从 Header 读取（Gateway 或上游服务传入）
        String traceId = req.getHeader(TraceIdConst.HEADER_TRACE_ID);
        if (!StringUtils.hasText(traceId)) {
            traceId = UUID.randomUUID().toString().replace("-", "");
        }

        // 2. 写入 MDC → Logback 日志自动携带
        MDC.put(TraceIdConst.MDC_TRACE_ID, traceId);
        MDC.put(TraceIdConst.MDC_SERVICE, serviceName);
        MDC.put(TraceIdConst.MDC_INSTANCE, getHostName());
        MDC.put(TraceIdConst.MDC_SOURCE, "http");

        // 3. 响应头回写（前端可用于关联排查）
        res.setHeader(TraceIdConst.HEADER_TRACE_ID, traceId);

        try {
            chain.doFilter(request, response);
        } finally {
            MDC.clear(); // ⚠️ 必须清理，防止线程池复用时 MDC 污染
        }
    }

    private String getHostName() {
        try { return InetAddress.getLocalHost().getHostName(); }
        catch (Exception e) { return "unknown"; }
    }
}
```

#### 7.4.4 MQTT TraceId 透传（发送端）

```java
// org/jeecg/modules/device/mqtt/TracingMqttPublisher.java
@Component
@RequiredArgsConstructor
public class TracingMqttPublisher {

    private final Tracer tracer; // Micrometer Tracer 自动注入

    /**
     * 构建携带 TraceId 的 MQTT 消息
     * 使用 MQTT 5.0 User Properties，不侵入业务 payload
     */
    public MqttMessage buildTracedMessage(byte[] payload, int qos) {
        MqttMessage message = new MqttMessage(payload);
        message.setQos(qos);

        Span currentSpan = tracer.currentSpan();
        if (currentSpan != null) {
            MqttProperties properties = new MqttProperties();
            properties.add(new MqttProperties.StringPair(
                MqttProperties.MqttPropertyType.USER_PROPERTY.value(),
                TraceIdConst.HEADER_TRACE_ID,
                currentSpan.context().traceId()));
            properties.add(new MqttProperties.StringPair(
                MqttProperties.MqttPropertyType.USER_PROPERTY.value(),
                TraceIdConst.HEADER_SPAN_ID,
                currentSpan.context().spanId()));
            message.setProperties(properties);
        }
        return message;
    }
}
```

#### 7.4.5 MQTT TraceId 透传（接收端）

```java
// MqttMessageDispatcher.java 修改 messageArrived() 方法
public void dispatch(String topic, MqttMessage message) {
    // 1. 从 User Properties 提取 traceId
    String traceId = extractTraceId(message);
    if (!StringUtils.hasText(traceId)) {
        traceId = "mqtt-" + UUID.randomUUID().toString().replace("-", "");
    }

    // 2. 写入 MDC
    MDC.put(TraceIdConst.MDC_TRACE_ID, traceId);
    MDC.put(TraceIdConst.MDC_SOURCE, "mqtt");
    MDC.put("mqttTopic", topic);

    try {
        log.info("MQTT消息到达 topic={}", topic);
        routeToHandler(topic, message);
    } finally {
        MDC.clear();
    }
}

private String extractTraceId(MqttMessage message) {
    MqttProperties props = message.getProperties();
    if (props == null) return null;
    // 遍历 User Properties 查找 X-Trace-Id
    List<MqttProperties.UserProperty> userProps =
        props.getProperties(MqttProperties.MqttPropertyType.USER_PROPERTY.value());
    if (userProps != null) {
        for (MqttProperties.UserProperty prop : userProps) {
            if (TraceIdConst.HEADER_TRACE_ID.equals(prop.value().getKey())) {
                return prop.value().getValue();
            }
        }
    }
    return null;
}
```

#### 7.4.6 AsyncMdcTaskDecorator

```java
// org/jeecg/common/trace/AsyncMdcTaskDecorator.java
public class AsyncMdcTaskDecorator implements TaskDecorator {
    @Override
    public Runnable decorate(Runnable runnable) {
        // 提交时：快照当前线程的 MDC
        Map<String, String> contextMap = MDC.getCopyOfContextMap();
        return () -> {
            try {
                // 执行时：恢复 MDC
                if (contextMap != null) {
                    MDC.setContextMap(contextMap);
                }
                runnable.run();
            } finally {
                MDC.clear();
            }
        };
    }
}
```

```
│  │
│  └─ 步骤4 [MQTT Handler] 在 iot-node-2 接收处理（跨 Pod！）
```

## 11. 性能优化与容量规划

### 11.1 性能影响评估

| 组件 | 影响方式 | 预估开销 | 缓解措施 |
|------|---------|---------|---------|
| Micrometer Tracing | 每个请求创建 Span 对象 | < 0.5 ms | 生产环境 10% 采样率 |
| MDC 读写 | ThreadLocal 操作 | < 0.01 ms | 可忽略 |
| loki4j Appender | 异步批量 HTTP 推送 | 不阻塞业务线程 | 内置异步队列 + 批量发送 |
| JSON 序列化 | logstash-logback-encoder | < 0.1 ms/条 | 比 Pattern Layout 多 ~30μs |
| Zipkin 上报 | 异步批量 HTTP 发送 | 不阻塞业务线程 | Brave AsyncReporter |

### 11.2 容量规划

假设：3 个服务节点，每节点 200 QPS，每请求 5 条日志。

| 指标 | 计算 | 结果 |
|------|------|------|
| 日志产生速率 | 3 × 200 × 5 | **3000 条/秒** |
| 每条日志大小 | JSON 平均 500 字节 | — |
| 日存储量 | 3000 × 500 × 86400 | **~130 GB/天**（未压缩） |
| Loki 压缩后 | 约 10:1 压缩比 | **~13 GB/天** |
| 30 天存储 | 13 × 30 | **~390 GB** |

> **建议**：生产环境配置 Loki 保留策略为 30 天，搭配对象存储（MinIO / S3）降低本地磁盘压力。

## 12. 实施计划

### 12.1 分阶段实施路线

| 阶段 | 内容 | 工作量 | 交付物 | 验收标准 |
|------|------|--------|--------|---------|
| **P0** 最小可用 | TraceIdMdcFilter + Logback JSON 改造 | 0.5 天 | 本地日志含 traceId | `grep traceId` 可在日志文件中找到 |
| **P1** HTTP 自动追踪 | 引入 Micrometer Tracing + Brave，配置采样率 | 0.5 天 | HTTP/Feign 自动插桩 | Zipkin UI 可见 HTTP Span |
| **P2** MQTT 链路贯通 | TracingMqttPublisher + Dispatcher 改造 + Gateway Filter | 1 天 | MQTT 跨 Pod 链路连续 | 两个 Pod 的 MQTT 日志 traceId 相同 |
| **P3** 日志聚合 | 部署 Loki + Grafana 数据源 + loki4j Appender | 1 天 | 集群日志统一查询 | Grafana 可按 traceId 查到所有节点日志 |
| **P4** 链路联动 | 部署 Zipkin + Grafana derivedFields 配置 | 0.5 天 | 日志 ↔ 链路一键跳转 | 点击日志中 TraceID 跳转 Zipkin |
| **P5** 补全覆盖 | Async/Job traceId 继承 + 日志脱敏 + 告警规则 | 0.5 天 | 全场景链路覆盖 | @Async 和 XXL-Job 日志含 traceId |

**总工作量**：约 **4 人天**。

## 13. 风险与应对措施

| # | 风险 | 影响 | 概率 | 应对措施 |
|---|------|------|------|---------|
| R1 | Loki 宕机导致日志丢失 | 中 | 低 | FILE_JSON Appender 本地兜底；Loki 配置持久化存储卷 |
| R2 | loki4j Appender 内存溢出 | 中 | 低 | 配置 `batchSize` 和 `sendQueueMaxBytes` 上限；监控 JVM 堆内存 |
| R3 | MQTT 3.1.1 不支持 User Properties | 高 | 中 | Fallback 到 payload 嵌入 `_traceCtx`；推动 EMQX 升级 MQTT 5.0 |
| R4 | MDC 线程池复用污染 | 高 | 中 | 所有 Filter / TaskDecorator / Dispatcher 在 finally 块清理 MDC；代码 Review 强制检查 |
| R5 | 日志量过大导致 Loki 存储压力 | 中 | 中 | 生产采样率 10%；Loki 配置 30 天保留策略；业务 DEBUG 仅开发环境启用 |
| R6 | Zipkin 内存存储重启丢失 | 低 | 中 | 当前为内存存储（适合中小规模）；后续可切换到 MySQL / Elasticsearch 持久化 |
| R7 | 定时任务链路缺失 | 低 | 低 | P5 阶段补全 XXL-Job Handler traceId 生成 |
| R8 | Gateway WebFlux MDC 误用 | 高 | 低 | 代码注释明确标注「WebFlux 禁用 MDC」；Code Review 检查项 |
