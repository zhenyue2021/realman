## 背景

ADR-0001（2026-06-30）确定了把 `realman-iot` 拆为设备管理中台、MQTT 集成平台、OTA 平台、瘦身 IoT 四个服务。此后产品侧输出了两份更完整的输入，暴露出 ADR-0001 三处需要修订的地方：

1. **业务架构 v1.2** 要求设备管理进一步拆两层——"设备信息基础服务"（SSOT，只读基础信息）与"设备管理业务平台"（注册/授权/四态/密钥/审计，写操作），而不是 ADR-0001 里笼统的一个"设备管理中台"。
2. **OTA V1.0.0 PRD** 把设备与后台的通信协议定义为**纯 HTTP**（心跳、Token 签发/续签/吊销、进度推送、状态补传均为 `POST /api/v1/...`），而不是 ADR-0001 假设的"MQTT 下发 + RocketMQ 事件转发"。
3. **业务架构 v1.2** 引入"平台能力总线"作为业务应用层与共享底座之间的显式分层，ADR-0001 没有这一层的治理设计。

此外，现状 `realman-iot` 与外部达尔文数采平台（数据处理模块）之间通过独立 RocketMQ Broker 桥接 4 条链路（OSS 鉴权、文件上报、设备状态、工单下发），设计初衷是"数采平台不能直连 MinIO、跨网络域"。SaaS 平台化之后数据处理模块已是平台内部的六大业务模块之一，这个前提不再成立。

## 决策

在 ADR-0001 基础上，追加以下四项决策：

### 1. 设备管理二次拆分为两层服务

| 服务 | 定位 | 读写特征 |
|---|---|---|
| 设备信息基础服务 `realman-device-info` | 设备基础信息 SSOT，供全部业务应用只读查询 | 读多写少 |
| 设备管理业务平台 `realman-device-mgmt` | 注册（`device_registration_secret` 生命周期）、Token 签发/续签/吊销、租户授权与隔离、四态测试标记、操作审计 | 低频写 + 强审计 |

设备管理业务平台是设备信息基础服务的写入方之一；其余业务应用只读 SSOT，不必接入完整的设备管理业务平台——这是支撑"能力可拆包售卖给第三方项目（如大洋电机）"的关键设计。

### 2. 设备通信中台南向协议维持 MQTT 不变，新增北向 HTTP 统一对外网关

**系统与设备之间的通信协议维持 MQTT 一种**（设备上电自注册接口除外，该接口在建立 MQTT 连接前以一次性 HTTP 调用完成）。新增的 HTTP 能力是**北向**——即通信中台对外统一输出的接口协议，服务两个目的：① 各业务服务（OTA、设备管理业务平台等）的对外 REST API，经由通信中台统一路由输出而非各自裸露端口；② 为 SmartArm 等**后续**不采用现有 MQTT SDK 的设备类型，预留一条可选的 HTTP 直连协议入口。

OTA PRD 第九章定义的心跳/Token/进度推送/状态补传接口契约，按此理解为：现有 MQTT 设备通过南向 MQTT Topic 传递同样语义的数据，通信中台归一化后按同一套契约转发给 OTA 平台；这套契约同时以 HTTP 形式在北向网关对外暴露。两侧共用同一套内部处理逻辑与状态机，只是南向/北向的传输介质不同——不需要为 MQTT 设备和 HTTP 设备各写一套业务逻辑。

### 3. 新增平台能力总线治理层

不引入新的物理中间件，而是由四件事组成的治理规范：Nacos 能力注册 + `*-contract` 模块作为机器可读能力目录 + 能力清单文档（人可读）+ Gateway 层统一鉴权与租户上下文（含 `X-Operator-Tenant-Id`）透传 + 契约版本治理（SemVer、废弃公告期）。

### 4. 数据处理模块（Darwin）桥接由 RocketMQ 切换为 HTTP 直连

现状 4 条 RocketMQ 链路（`daily_GLN_PLATFORM` 主题下的 OSS 鉴权申请/响应、文件地址上报、设备状态推送、工单下发）全部改为设备通信中台与数据处理模块之间的同步 HTTP 调用。采用双写过渡（功能开关控制走 MQ 还是 HTTP），灰度验证后下线 RocketMQ 生产者/消费者代码。平台内部仍可保留事件总线能力，但仅用于真正需要削峰/多消费者广播的场景，不再用于"同域内两个业务模块之间"的桥接。

## 备选方案

### 方案 A：维持 ADR-0001 的"设备管理中台"不再二次拆分

- **优点**：少一次服务拆分，短期改动更小。
- **缺点**：无法支撑"第三方项目只接入基础信息查询、不接入完整业务操作"的产品化诉求；SSOT 高频只读查询与审计类写操作的发布节奏被迫绑定。
- **结论**：不采纳；业务架构 v1.2 已明确要求两层。

### 方案 B：现有设备协议切到 HTTP，对齐 OTA PRD 字面描述（初版曾采纳，现已否决）

- **优点**：与 OTA V1.0.0 PRD 第九章的字面描述（`调用方：设备端 SDK`）完全一致，不需要做协议等价映射。
- **缺点**：要求现有 master/slave 设备 SDK 从 MQTT 改为 HTTP，改造成本和风险远超预期；与"设备通信中台是唯一通道、协议变更不影响业务"的初衷相悖——协议本身发生了大改动。
- **结论**：不采纳。确认 PRD 第九章的 HTTP 接口是"逻辑契约"，现网设备侧仍用 MQTT 承载同等语义，HTTP 契约作为通信中台北向网关对外暴露的等价形式，服务外部系统与未来 HTTP 原生设备（SmartArm）。

### 方案 C：数据处理桥接保留 RocketMQ，仅新增能力总线治理

- **优点**：改动面更小，不涉及 Darwin 团队配合改造。
- **缺点**：与"SaaS 平台化后同域内不再需要跨系统异步解耦"的既定简化方向不符，继续背负独立 RocketMQ Broker 的运维成本（NameServer 互联、DLQ 告警等）。
- **结论**：不采纳；按第六章方案退役。

## 后果

### 正面

- 设备基础信息可作为独立、轻量的能力对外（第三方项目）单独提供，不必绑定完整设备管理业务。
- 现有设备协议不受影响（继续 MQTT），同时通过北向 HTTP 网关满足 OTA PRD 的契约要求，并为 SmartArm 等后续项目预留标准化接入点，避免"文档一套、实现一套"的长期维护分歧。
- 去掉数据处理的 RocketMQ 桥接后，减少一套独立中间件的运维面（Broker、NameServer、DLQ 告警），链路延迟也从"异步排队"降为"同步调用"。
- 能力总线治理层落地后，为后续更多第三方接入/私有化拆包提供统一规范。

### 负面 / 代价

- 相较 ADR-0001，微服务数量进一步增加（设备管理拆两层），运维复杂度上升。
- OTA 平台需要按 PRD 重建状态机、密钥管理、错误码体系，工作量远超"代码搬家"，Phase 1 排期需要相应延长（详见设计文档第八章）。
- 数据处理 HTTP 化需要外部团队（Darwin）配合改造调用方式，存在协调风险。
- 设备管理引入租户隔离是存量数据的结构性变更，需要单独的数据迁移方案。
