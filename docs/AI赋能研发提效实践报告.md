# AI 赋能 -  IoT 平台单机架构向集群架构演进实践

---

## 一、背景与项目概述

### 1.1 项目背景

realman-boot-iot 是一套面向工业机器人远程遥操场景的物联网平台，核心功能包括：

- 主控设备（遥操终端）与机器人设备的 MQTT 双向通信
- 设备上下线感知与实时状态同步
- 遥操会话管理（工单驱动）
- 实时视频流与 WebSocket 推送 等

### 1.2 演进背景

单实例模式下能够保证开发及测试环境下的功能完整，但正式上线后，随着接入设备数量增长和业务并发压力提升，单实例部署已无法满足高可用与弹性伸缩需求，需将服务改造为多实例集群架构。然而，IoT 场景中大量依赖**长连接**（MQTT、WebSocket）和**内存状态**（遥操关系、设备在线感知），这使得集群化改造比普通业务服务复杂得多——每一项长连接机制的集群化都是独立的技术难题。

| 维度 | 单机阶段 | 集群目标 |
|------|---------|---------|
| 部署方式 | 单 JAR 进程 | Spring Cloud 微服务多实例 |
| MQTT 消费 | 单连接全量消费 | 多实例负载均衡消费，消息不重复 |
| WebSocket 推送 | 单节点本地推送 | 跨节点消息路由，连接无需感知落在哪台实例 |
| 运行时状态 | 内存 Map / 轮询 DB | Redis 集中化，任意实例可读写 |
| 遥操关系绑定 | 无持久化，重启即丢 | 双向缓存，支持正反查，生命周期可控 |

## 二、核心改造难点

### 2.1 MQTT 消息重复消费问题

#### 难点说明

MQTT 协议本身不区分"消费者组"，所有订阅了同一 Topic 的客户端都会收到相同的消息。单机时只有一个服务实例，天然不存在重复；但扩容至多实例后，每个实例都与 EMQX Broker 建立独立连接并订阅相同 Topic，同一条设备消息会被所有实例**同时收到**，导致重复写库、重复推送、重复触发告警等连锁问题。

这是 MQTT 与 Kafka/RocketMQ 等消息队列最大的架构差异——消息队列天然支持消费者组语义，而 MQTT 不支持。直接替换 MQTT 为消息队列成本极高，且会丢失 MQTT 在 IoT 设备侧的协议优势。

#### 传统解决方式

- **方案一：分布式锁**。每个实例收到消息后先抢锁，抢不到则丢弃。需要引入 Redis 分布式锁，且存在锁超时、消息丢失等边界问题，逻辑复杂。
- **方案二：消息去重表**。将消息 ID 写入 Redis Set，收到消息先查重再处理。需要保证消息 ID 的全局唯一性，且 Topic 设计不统一时难以提取标准 ID。
- **方案三：单实例消费 + 内部分发**。固定一个实例做 MQTT 消费，再通过内部 MQ 分发给其他实例处理业务。引入了额外中间件，架构复杂度显著上升。

上述方案均需要较多额外开发工作量，且各有边界问题，评估和选型本身就需要耗费大量精力。

#### AI 辅助方式

描述集群化诉求后，AI 直接指出 EMQX 本身已支持**共享订阅**（`$share`）机制——同一共享组内的多个订阅者，Broker 侧保证同一条消息只投递给组内一个成员，天然实现负载均衡，无需在应用侧做任何去重逻辑。

AI 进一步给出了改造范围最小化的方案：在订阅 Topic 字符串前拼接共享组前缀，业务处理层零改动；同时结合 Nacos 配置中心，通过开关控制是否启用共享订阅，方便灰度验证和回滚。

整个方案的评估、选型、实现过程中无需查阅大量文档，AI 将关键知识点（EMQX 共享订阅语法、与普通订阅的兼容性、热扩容行为）一并说明，开发者只需确认方案合理性后直接落地。

### 2.2 WebSocket 跨实例推送问题

#### 难点说明

WebSocket 是有状态的长连接，前端建立连接后，该 Session 对象只存在于**接收连接的那个服务实例**的内存中。当设备消息被实例 A 的 MQTT Handler 消费后，若前端的 WebSocket 连接恰好在实例 B 上，实例 A 无法找到该 Session，消息推送直接丢失。

这一问题在单机时完全不存在，却是集群化后必须首先解决的核心问题之一。且由于 WebSocket 的长连接特性，无法像 HTTP 接口那样简单地通过负载均衡转发解决——即使配置了会话粘性（sticky session），MQTT 消费实例与 WebSocket 实例不一致的问题依然存在。

#### 传统解决方式

- **引入消息中间件（如 RocketMQ/Kafka）**：MQTT Handler 消费后发布到 MQ，每个实例订阅 MQ，本地有该 Session 则推送。引入了新中间件，运维成本上升，且需要保证 MQ 消费的幂等性。
- **会话中心化（如 Netty + Redis 存储 Session 映射）**：统一管理 WebSocket Session 与实例的映射关系，需要定制 WebSocket 框架，改造量大。
- **强制会话粘性**：通过负载均衡器将同一 deviceCode 的请求路由到固定实例。配置复杂，且实例故障时会话直接丢失，无法平滑切换。

#### AI 辅助方式

AI 识别到项目已引入 Redis，直接建议使用 **Redis Pub/Sub** 作为轻量级跨实例消息总线，无需引入新中间件。方案思路清晰：MQTT Handler 收到设备消息后向 Redis 特定 Channel 发布消息，所有服务实例均监听该 Channel，各自检查本地是否持有对应 Session，持有则推送，不持有则忽略。

AI 同时指出该方案的一个边界场景：若 Redis Pub/Sub 消息在推送时前端恰好断连，需要结合重连机制或消息补偿，并给出了处理建议。这类边界分析在传统开发中通常需要联调阶段才能暴露。

### 2.3 遥操关系的运行时状态管理

#### 难点说明

遥操场景中，主控设备与机器人设备之间存在**动态绑定关系**：主控登录并关联工单后，平台需要实时知道"某主控当前控制哪台机器人"，机器人上报状态时也需要反查"当前绑定的主控是谁"，以便精准推送给对应前端。

单机阶段这种关系存于内存，访问极快且无需额外设计。集群化后内存不共享，若仍用内存，每个实例只能看到自己建立的绑定关系，跨实例查询完全失效。若改为每次查数据库，则在高频的设备状态上报场景下（每秒数十次），数据库压力极大。

更复杂的是，这种绑定关系有明确的生命周期——遥操开始时建立、停止时清除——若处理不当，脏数据会导致消息推送到错误的前端。

#### 传统解决方式

- **每次查库**：实现简单，但高频查询场景下数据库成为瓶颈，且需要设计专门的关系表和清理逻辑。
- **内存 + 定时同步**：各实例维护本地缓存，定时从 DB 同步。存在数据不一致窗口期，且遥操关系的实时性要求较高，窗口期内可能推送到错误实例。
- **引入分布式缓存（手工设计）**：需要自行设计 Key 命名规范、正反查结构、失效策略、清理时机，全靠经验积累，容易遗漏边界情况。

#### AI 辅助方式

AI 在理解遥操业务流程后，直接给出了双向 Key 设计方案（主控→机器人、机器人→主控各一个 Key），并明确了写入时机（主控登录解析出工单机器人时）和清除时机（停止遥操时），同时建议添加安全 TTL 作为兜底，防止因异常流程导致 Key 永不过期产生脏数据。

这种"双向索引 + 生命周期精确管理"的设计思路，传统开发中需要有经验的架构师才能在第一时间给出，通常在联调或生产问题暴露后才会补充反查方向。

### 2.4 消息处理层的可维护性问题

#### 难点说明

IoT 平台的 Topic 体系随业务迭代持续膨胀，当前已有 15+ 种不同语义的 Topic（设备状态上报、指令 ACK、OTA 进度、操作日志、摄像头流、SLAM 地图、设备上下线等），且主控设备与机器人设备的 Topic 格式也不统一（`device/{code}/...` 与 `{code}/master/...` 并存）。

若所有消息处理逻辑集中在一个入口函数中，随着分支数量增长，代码将变成难以维护的"巨型 if-else"。每次新增 Topic 类型都需要修改核心类，改错一行可能影响所有消息处理，测试范围难以界定。

#### 传统解决方式

大多数项目在初期缺乏规划，直接在 MQTT 回调中堆砌 if-else，随着 Topic 种类增加逐渐失控。重构时需要梳理现有所有 Topic 的处理逻辑，拆分风险高，通常只能"下次再说"，形成技术债。

若要在初期就做好分层设计，需要开发者有较强的架构意识，且需要花费时间设计分发规则、Handler 接口规范、路由匹配逻辑，在需求压力下这部分工作往往被压缩。

#### AI 辅助方式

AI 在理解当前 Topic 结构后，直接给出了"分发器 + Handler"的分层方案——一个统一路由层只负责按 Topic 模式匹配后转发，各 Handler 只处理自己关心的业务逻辑，互不感知。

AI 同时识别出项目中两种 Topic 格式并存的问题，给出了正则分段匹配的路由策略，覆盖了 `device/` 前缀、`master/` 前缀、`{code}/master/` 格式、`$SYS/` 系统事件等全部情况，避免了手工设计时容易遗漏格式分支的问题。

新增 Topic 类型的改动范围被压缩到最小：只需新增一个 Handler 类，并在路由层加一行分支，不影响任何现有 Handler 的逻辑。

### 2.5 其他典型难点

| 改造点 | 难点描述 | 传统方式 | AI 辅助方式 |
|--------|---------|---------|-----------|
| **设备在线状态感知** | 集群下内存标记失效，轮询 DB 实时性差且有压力 | 定时任务轮询数据库更新在线状态，有延迟 | 指出 Redis Key TTL 自动失效机制，Key 消失即表示设备离线，实时且零轮询 |
| **指令下发等待 ACK** | 下发指令后需等待设备回包，同步阻塞或轮询均有问题 | 轮询 DB 等待状态字段变更，或 Thread.sleep 定时检查 | 引入 CompletableFuture + Redis Pub/Sub，异步等待，超时可精确控制 |
| **日志未打印的行为分析** | MQTT ACL 回调日志存在，但消息处理日志缺失，表象像消息丢失 | 加日志重启，等设备重新触发，排查周期长 | 从日志中的 `action=subscribe` 直接判断为订阅动作而非发布，无需重启即可定位 |

### 3.3 典型高价值场景还原

**场景一：集群重复消费 —— 架构阶段识别，避免上线后返工**

- 传统路径：单机功能验证通过 → 直接扩容 → 压测或生产发现重复写库 → 紧急排查 → 引入分布式锁或去重表 → 回归测试 → 再次上线。整个返工链路涉及多个阶段，修复成本是设计阶段的 5~10 倍。
- AI 路径：描述"要做集群化"时，AI 主动指出 MQTT 订阅模型与消费者组语义的差异，在架构评审阶段即给出共享订阅方案，问题在落笔写代码前已解决。

**场景二：WebSocket 跨实例推送 —— 无需引入新中间件**

- 传统路径：通常第一反应是引入 RocketMQ 或 Kafka 作为消息总线，需要申请中间件资源、接入 SDK、处理消费幂等，评估周期长，运维成本增加。
- AI 路径：识别到项目已有 Redis 依赖，直接建议复用 Redis Pub/Sub，方案轻量，无新增中间件，改造范围精确控制在推送层。

**场景三：日志缺失排查 —— 无需重启即可定位**

- 传统路径：发现消息处理日志没有打印，通常会怀疑消息丢失、Handler 未注册、Bean 初始化失败等，逐一加日志重启验证，排查周期 30~60 分钟。
- AI 路径：提供 ACL 回调日志后，AI 直接从 `action=subscribe` 字段判断这是设备**订阅**动作而非**发布**，消息处理日志本不应出现，排查过程无需重启，2 分钟内给出结论。

**场景四：级联编译错误 —— 直指根因，不绕弯路**

- 传统路径：编译报错"程序包 entity 不存在、mapper 不存在"，开发者倾向于先检查类文件是否缺失、包路径是否正确，排查多个具体类后才可能想到是上游模块未安装的级联问题。
- AI 路径：根据"同一模块内的类互相找不到"这一异常特征，判断根因是上游依赖模块（`realman-boot-base-core`）未安装到本地 Maven 仓库，直接给出最小构建命令，10 分钟内解决。

## 四、AI 辅助开发模式总结

### 4.1 协作流程

在本次改造中，AI 介入的方式并非简单的"给我写代码"，而是形成了一套完整的协作闭环：

1. **需求描述阶段**：开发者用自然语言描述业务场景和改造目标，AI 输出技术方案选型建议及关键风险点，开发者确认后进入实现阶段。
2. **实现阶段**：AI 输出精确到文件和行的改动建议，开发者审核业务合理性后落地；遇到报错或行为异常，直接将日志/错误信息交给 AI 分析。
3. **收尾阶段**：AI 同步输出变更说明和技术文档，无需额外投入文档工时。

### 4.2 AI 的能力边界

AI 的价值集中在**知识密集型、模式化程度高**的工作上；业务决策和生产风险判断仍需人工主导。

| AI 擅长 | 仍需人工判断 |
|--------|------------|
| 已知技术模式的快速套用（Redis、MQTT、Spring Cloud 等） | 业务规则的合理性（如遥操关系的生命周期由业务决定） |
| 编译 / 运行时报错的根因定位 | 与外部团队（硬件、固件侧）的协议协商 |
| 架构设计中常见陷阱的识别与规避 | 生产环境的变更窗口与回滚预案 |
| 代码分层与模块边界设计 | 性能基准数据的实测与容量规划 |
| 文档、注释、对比报告的生成 | 需求优先级与资源排期的决策 |

### 4.3 使用建议

1. **架构设计前置 AI 咨询**：新模块设计阶段先与 AI 探讨集群化、并发、缓存一致性等潜在问题，将风险消灭在评审阶段。
2. **故障排查优先使用 AI**：日志分析、编译错误、依赖问题等，AI 的定位速度显著优于人工逐步排查，尤其适合经验较少的成员。
3. **编码前用 AI 做方案预演**：描述技术方案后让 AI 指出边界场景，可有效减少联调阶段才暴露问题的情况。
4. **文档与开发同步进行**：避免"开发完成后补文档"的拖延，将文档成本接近清零。

## 五、总结

本次 IoT 平台从单机到集群的架构演进，涉及 MQTT 共享订阅、WebSocket 跨节点推送、Redis 状态外化、遥操关系缓存等多个技术难点，每一项在传统模式下都需要经过调研→方案评审→实现→联调的完整周期。

在 AI 全程辅助下，核心改造模块的研发周期从预估 **25.5 人天** 压缩至实际 **4.75 人天**，节省约 **81%** 的开发工时。更重要的是，多个集群化陷阱在架构阶段即被识别并规避，减少了联调和返工成本，整体质量显著提升。

AI 的核心价值可以归纳为三点：
- **把风险前移**：将联调阶段才暴露的架构问题提前到设计阶段解决
- **压缩知识检索成本**：开发者无需花大量时间查阅文档和技术博客，AI 直接给出经过筛选的可落地方案
- **让开发者聚焦于判断而非查找**：从"找答案"转变为"判断答案是否正确"，专注于真正需要业务理解的部分
