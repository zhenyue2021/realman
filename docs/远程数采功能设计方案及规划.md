# 遥操平台远程数采实现方案（IOT 模块重构版）

> 目标：严格基于 `realman-boot-iot` 现有模块实现遥操平台远程数采能力；**不参考 darwin 包**；工单管理暂不纳入本次范围。

## 1. 范围界定

### 1.1 本次实现范围
- 遥操会话开始/结束编排（主控 + 机器人）
- 遥操期间核心实时数据采集与透传（机器人状态、操作日志、关键参数）
- MQTT 指令与 ACK 可靠性保障（超时、幂等、补偿）
- WebSocket 实时推送给遥操平台
- 数据落库（操作记录、命令记录、状态快照）

### 1.2 暂不实现
- 工单管理全链路（创建、流转、审计、超时策略）
- 与工单绑定的流程控制（后续按新逻辑单独接入）

---

## 2. 现有 IOT 模块能力盘点（可复用）

### 2.1 编排与设备域服务
- `IotDeviceTeleopService`：已有开始/结束遥操主流程、WebRTC 开停。
- `IotDeviceSupport`：设备存在性/类型校验。
- `IotDeviceStatusQueryService`：设备状态查询能力。

### 2.2 协议与消息层
- `MqttPublisher`：统一 MQTT 下发。
- `MqttMessageDispatcher` + `handler/*`：按 Topic 路由上行消息。
- `DeviceWebSocketServer` / `DeviceWebSocketChannelHandler`：WebSocket 推送通道。

### 2.3 数据持久化
- `IotDeviceCommandRecordServiceImpl`：命令下发记录。
- `DeviceOperationLogServiceImpl`：操作日志。
- `IotDeviceStatusMapper`：状态存储。

---

## 3. 新增/重构目标能力

### 3.1 远程数采会话模型（新增）
新增“遥操数采会话”概念，解耦工单：
- 唯一会话ID（sessionId）
- controllerCode / robotCode
- operator / startTime / endTime
- 状态：INIT / RUNNING / STOPPING / CLOSED / FAILED

### 3.2 数据采集通道分层（新增）
1. **实时层**：MQTT 上行 -> Dispatcher -> WebSocket 推送。
2. **快照层**：固定频率（如 1s/5s）落库关键状态（位置、电量、速度、告警）。
3. **事件层**：异常、急停、重连、ACK 超时等事件持久化。

### 3.3 ACK 与超时机制（重构）
- 保留现有 PendingService 思路，统一到 `TeleopSessionPendingService`。
- 下发命令必须携带 `commandId/sessionId`。
- 超时后统一失败态回写 + 终止会话补偿。

### 3.4 无工单模式适配（重构）
- 现有 stop/start payload 中的 `workOrderId` 以 `sessionId` 替代。
- 工单字段置为可空，避免强依赖。

---

## 4. 具体落地计划（按迭代执行）

## 迭代A：会话主链路打通（优先）
1. 新增实体 `IotTeleopSession` + Mapper + Service。
2. `startTeleopNoStream/stopTeleopByCode` 改造：
   - 创建/关闭 session
   - payload 补充 `sessionId`
   - 记录命令与事件
3. 新增 API：
   - `POST /teleop/session/start`
   - `POST /teleop/session/stop`
   - `GET /teleop/session/{sessionId}`

**验收标准**
- 可在无工单前提下完成开始/结束遥操。
- 会话状态与设备占用状态一致。

## 迭代B：实时数采透传与落库
1. 在 `RobotSlaveStatusHandler` / `DeviceStatusHandler` 增加 session 关联。
2. 新增 `TeleopDataCollectorService`：
   - 汇聚状态上报
   - 定频快照落库
3. WebSocket 推送 payload 统一：`sessionId + type + data + ts`

**验收标准**
- 平台可实时收到状态流。
- 数据库可按 sessionId 查询全程关键状态。

## 迭代C：可靠性与可观测
1. 增加超时任务清理（长时间无心跳自动 STOPPING）。
2. 增加告警事件：ACK timeout、device offline、webrtc start fail。
3. 增加运维查询接口：按 sessionId 拉取事件时间线。

**验收标准**
- 异常场景可自动收敛并可追溯。

## 5. 代码改造点清单（精确到模块）

### 5.1 `realman-boot-iot-biz`
- `service/impl/device/IotDeviceTeleopService.java`：会话化改造（去工单硬编码）。
- `mqtt/handler/RobotSlaveStatusHandler.java`：状态事件挂接 session。
- `mqtt/handler/DeviceCommandAckHandler.java`：ACK -> session pending。
- 新增 `service/TeleopSessionService`、`service/TeleopDataCollectorService`。
- 新增 entity/mapper：`IotTeleopSession`、`IotTeleopSnapshot`、`IotTeleopEvent`。

### 5.2 `realman-boot-iot-api`
- 新增 `controller/TeleopSessionController`。
- 新增 DTO：start/stop/query/request/response。

### 5.3 `sql`
- 增量 SQL：3 张表（session/snapshot/event），全部按 `session_id` 建索引。

---

## 6. 开发约束
- 严禁引用 darwin 包实现。
- 工单相关 service/controller/mapper 不改业务行为，仅解除耦合点。
- 所有新增逻辑必须带单测：
  - session 状态流转
  - ACK 超时
  - 设备离线终止

---

## 7. 下一步执行顺序（落代码）
1. 先完成迭代A最小可用版本并联调 MQTT topic。
2. 再做迭代B的数据采集落库与推送。
3. 最后补齐迭代C可靠性与运维接口。

> 说明：以上方案已按“工单暂不做、逻辑后续会变更”的要求拆分，保证先把遥操平台远程数采主能力独立上线。
