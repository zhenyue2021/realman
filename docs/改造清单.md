# 集群部署改造清单

基于对项目的深入分析，以下是集群部署改造清单，按优先级排列。

---

## 🔴 P0 - 必须立即修复

### 1. ✅ Session 分布式存储（已天然实现，无需改造）

- **位置**：`ShiroConfig.java`、`ShiroRealm.java`
- **核查结论**：原描述有误，项目采用 **JWT 无状态架构**，不存在服务端 Session 问题：
  - `ShiroConfig:239` `setSessionStorageEnabled(false)` — Shiro HTTP Session 已主动关闭
  - `ShiroRealm:201` Token 有效性校验查 Redis（`PREFIX_USER_TOKEN + token`）
  - `ShiroRealm:212` Token 续期写 Redis，TTL 自动延续
  - `ShiroConfig:283` 权限/角色缓存使用 `RedisCacheManager`，已在 Redis
  - `ShiroConfig:309-317` Sentinel 模式已实现，配置 `spring.data.redis.sentinel` 即自动切换
- **集群切换操作**：仅需在 Nacos 中将 Redis 连接改为 Sentinel 配置，代码零改动
  ```yaml
  spring:
    data:
      redis:
        sentinel:
          master: mymaster
          nodes:
            - 172.16.2.10:26379
            - 172.16.2.11:26379
            - 172.16.3.10:26379
        password: your_strong_redis_password
  ```
- **⚠️ 勿引入 `spring-session-data-redis`**：与当前无状态架构冲突，引入会破坏现有鉴权流程

### 2. ✅ 短信限流本地 Map（已完成 Redis 改造）

- **位置**：`DySmsLimit.java`
- **改造内容**：
  - 移除三个 `static ConcurrentHashMap`，改为 `@Component` 注入 `RedisUtil`
  - 计数器：`sms:limit:count:{ip}`，TTL 60s，`INCR` 原子自增，窗口过期自动归零
  - 黑名单：`sms:limit:blacklist:{ip}`，TTL 86400s（24 小时后自动解除，优于原版永久黑名单）
  - 三处调用方（`LoginController`、`SysUserController`、`SysUserServiceImpl`）均改为注入 Bean

## 🟠 P1 - 本周完成

### 4. `@Scheduled` 定时任务重复执行

- **位置**：
    - `QqWryDatHotReloadScheduler.java:18` 无分布式锁
    - `RedisServiceImpl.java:149` 监控记录无锁
    - `SignalingKeyScheduler.java:78` ✅ 已有锁（可参考）
- **方案**：统一参考 `SignalingKeyScheduler` 的 `setIfAbsent` 锁模式；或迁移到已有的 Quartz（`isClustered: true` 已配置好）/ XXL-Job

### 5. `@Cacheable` 本地缓存

- **位置**：12+ 个 Service（SysDepart、SysUser 等）
- **后果**：Node1 `@CacheEvict` 后，其他节点本地缓存仍脏
- **方案**：`application.yml` 添加 `spring.cache.type: redis`

### 6. Redis 监控指标本地 Map

- **位置**：`RedisServiceImpl.java:42` `static HashMap REDIS_METRICS`
- **方案**：迁到 Redis 存储 + 聚合查询

## 🟡 P2 - 本月完成

### 7. WebSocket 节流本地状态

- **位置**：`DeviceWebSocketServer.java:66-82` 节流 `latestMessages` 是 static
- **说明**：消息推送主通道已通过 Redis Pub/Sub 广播实现集群安全 ✅，但节流状态各节点独立
- **方案**：节流缓存迁到 Redis 短 TTL

### 8. 用户导入进度本地缓存

- **位置**：`ImportSysUserCache.java:13` static HashMap 存进度
- **后果**：导入发生在 Node1，前端轮询进度路由到 Node2 显示 0%
- **方案**：迁到 Redis，30 分钟 TTL

### 9. Quartz instanceId

- **位置**：`application-dev.yml` `instanceId: AUTO`
- **方案**：改为 `${HOSTNAME:node-${random.uuid}}` 防碰撞

### 10. MyBatis-Plus ID 策略

- **方案**：确认主键使用 `IdType.ASSIGN_ID`（雪花算法），避免数据库自增

## 部署层面补充

1. **Nginx / 负载均衡**：WebSocket 需开启 sticky session 或确认 Redis Pub/Sub 广播覆盖
2. **日志聚合**：已有 Loki / Zipkin，集群下务必带 `${HOSTNAME}` 区分节点
3. **EMQX**：建议集群部署，客户端连接串配多节点
4. **MinIO**：生产用分布式模式，避免单点
