## 🔴 P0 - 必须立即修复

### 1. ✅ Session 分布式存储（已天然实现，无需改造）

- **位置**：`ShiroConfig.java`、`ShiroRealm.java`
- **核查结论**：原描述有误，项目采用 **JWT 无状态架构**，不存在服务端 Session 问题：
  - `ShiroConfig:239` `setSessionStorageEnabled(false)` — Shiro HTTP Session 已主动关闭
  - `ShiroRealm:201` Token 有效性校验查 Redis（`PREFIX_USER_TOKEN + token`）
  - `ShiroRealm:212` Token 续期写 Redis，TTL 自动延续
  - `ShiroConfig:283` 权限/角色缓存使用 `RedisCacheManager`，已在 Redis
  - `ShiroConfig:309-317` Sentinel 模式已实现，配置 `spring.data.redis.sentinel` 即自动切换
- **集群切换操作**：仅需在 Nacos 中将 Redis 连接改为 Sentinel 配置，代码零改动
  ```yaml
  spring:
    data:
      redis:
        sentinel:
          master: mymaster
          nodes:
            - 172.16.2.10:26379
            - 172.16.2.11:26379
            - 172.16.3.10:26379
        password: your_strong_redis_password
  ```
- **⚠️ 勿引入 `spring-session-data-redis`**：与当前无状态架构冲突，引入会破坏现有鉴权流程

### 2. ✅ 短信限流本地 Map（已完成 Redis 改造）

- **位置**：`DySmsLimit.java`
- **改造内容**：
  - 移除三个 `static ConcurrentHashMap`，改为 `@Component` 注入 `RedisUtil`
  - 计数器：`sms:limit:count:{ip}`，TTL 60s，`INCR` 原子自增，窗口过期自动归零
  - 黑名单：`sms:limit:blacklist:{ip}`，TTL 86400s（24 小时后自动解除，优于原版永久黑名单）
  - 三处调用方（`LoginController`、`SysUserController`、`SysUserServiceImpl`）均改为注入 Bean

### 5. ✅ `@Cacheable` 本地缓存（已核查，已配置 TTL）

- **核查结论**：原描述有误，当前**已天然使用 Redis** 作为 Spring Cache 后端，`@CacheEvict` 跨节点即时生效：
  - `jeecg-boot-common` 中的 `RedisConfig` 类自带 `@EnableCaching` + `@Bean cacheManager()`，提供 `RedisConfigCacheManager extends RedisCacheManager`
  - 该 Bean 通过 `org.jeecg` 包扫描自动注册，项目无需额外配置 `spring.cache.type: redis`
  - `@CacheEvict` 删除的是 Redis Key，集群内所有节点下次访问都会重新加载数据库
- **实际改造**：各缓存默认 TTL=0（永不过期），直接 DB 改数据后无法自动失效。已在 `application-dev.yml` 中通过 `jeecg.redis.cache-ttls` 配置合理 TTL：

  | 缓存常量 | Redis Key 前缀 | TTL |
  |---|---|---|
  | `SYS_USERS_CACHE` | `sys:cache:encrypt:user` | 1800s |
  | `SYS_DICT_CACHE` | `sys:cache:dict` | 3600s |
  | `SYS_ENABLE_DICT_CACHE` | `sys:cache:dictEnable` | 3600s |
  | `SYS_DICT_TABLE_CACHE` | `sys:cache:dictTable` | 3600s |
  | `SYS_DATA_PERMISSIONS_CACHE` | `sys:cache:permission:datarules` | 3600s |
  | `SYS_DEPARTS_CACHE` | `sys:cache:depart:alldata` | 3600s |
  | `SYS_DEPART_IDS_CACHE` | `sys:cache:depart:allids` | 3600s |
  | `DefIndexConst.CACHE_KEY` | `sys:cache:def_index` | 86400s |
  | `SYS_USER_ID_MAPPING_CACHE` | `sys:cache:user:id_mapping` | 3600s |

- **⚠️ 生产环境操作**：将 `jeecg.redis.cache-ttls` 块迁移到 Nacos，支持不重启热调整 TTL

### 6. ✅ Redis 监控指标本地 Map（已完成 Redis 改造）

- **位置**：`RedisServiceImpl.java`
- **改造内容**：
  - 移除 `static HashMap REDIS_METRICS`，不再使用 JVM 本地存储
  - 引入两个 Redis ZSET 键：`monitor:redis:metrics:dbSize` / `monitor:redis:metrics:memory`
  - score = 采集时间戳（毫秒），天然按时间升序，支持范围裁剪
  - `recordCustomMetric()`：加锁采集后调用 `writeMetricsToZSet()`，ZADD → 裁剪超出60条的旧数据 → 刷新 TTL（2小时）
  - `getMetricsHistory()`：调用 `readMetricsFromZSet()` 从 Redis 读，JSON 反序列化回 `Map`，返回格式与原接口兼容
  - Controller / 前端零改动
- **集群效果**：分布式锁保证只有一个节点写入，数据集中存 Redis，所有节点查询返回一致结果

### 7. ✅ WebSocket 节流本地状态（已完成改造）

- **位置**：`DeviceWebSocketServer.java`
- **改造内容**：
  - `latestMessages`（static ConcurrentHashMap）保留作本地去重缓冲，不迁 Redis（节流目的是 100ms 内丢弃重复帧，本地比较合适）
  - **根本问题**：原 static 块的调度器 flush 时直接调 `send()`（本地推送），高频消息完全绕过 Redis Pub/Sub，跨节点客户端收不到消息
  - **修复**：移除 static 块，增加 `@PostConstruct startThrottleScheduler()`，flush 时改调 `instance.redisPublish()`，消息走 Redis Pub/Sub 广播路径
  - 新增 `private static volatile DeviceWebSocketServer instance`，`@PostConstruct` 中赋值，供调度器线程访问实例方法
  - 新增 `import jakarta.annotation.PostConstruct`
- **集群效果**：
  - 节流去重：在 MQTT 接收节点本地完成（正确，仅该节点有该消息）
  - 跨节点推送：flush → `redisPublish()` → Redis Pub/Sub → 所有节点 `onMessage()` → 各自 `send()` 本地 sessions
  - 涵盖 `pushRobotStatus` / `pushMasterStatus` / `pushMasterCmdStatus` 三个高频推送方法

## SaaS 服务化改造（Realman SaaS 服务平台）

> 目标：将现有项目从“单体业务系统”升级为“多租户 + 插件化 + 可运营”的 SaaS 平台。以 `system` 为基座，设备管理、数据采集、远程升级、数据大屏作为可授权插件服务。

### 一、总体目标架构

- **平台基座（System Core）**
  - 统一身份认证（注册/登录/SSO）
  - 统一租户管理（租户、组织、成员、角色）
  - 统一授权计费（套餐、订阅、License、用量）
  - 统一运维能力（审计、监控、告警、日志）
- **插件服务层（Plugin Services）**
  - 设备管理服务（Device Management）
  - 数据采集服务（Data Collection / Ingestion）
  - 远程升级服务（OTA / FOTA）
  - 数据大屏服务（Dashboard / BI）
- **平台公共能力层（Shared Capabilities）**
  - API 网关、配置中心、消息总线、对象存储、任务调度、规则引擎、通知中心

### 二、技术改造点（按优先级）

#### P0：SaaS 最小闭环（必须优先）

1. **多租户模型落地（数据隔离）**
   - 改造点：核心业务表统一引入 `tenant_id`，查询链路强制注入租户条件。
   - 实现方案：
     - MyBatis-Plus 增加租户拦截器 + 白名单表机制（系统字典等公共表可共享）。
     - 新增租户上下文（从 JWT / 网关注入）并在 DAO 层自动透传。
     - 对历史数据执行“一次性迁移脚本”，补齐默认租户归属。
   - 验收标准：A 租户无法查询/修改 B 租户任何业务数据。

2. **统一认证与授权中心（System 作为基座）**
   - 改造点：把“用户注册、企业入驻、成员邀请、角色授权”统一收敛到 `system` 模块。
   - 实现方案：
     - 用户身份（User）与租户成员关系（TenantUserRel）解耦。
     - RBAC 升级为 `租户级 RBAC + 插件权限点`（如 `device:read`, `ota:publish`）。
     - JWT 增加 `tenant_id`、`plan_code`、`enabled_plugins` 等声明。
   - 验收标准：同一用户可切换多个租户；租户内权限最小化生效。

3. **插件授权开关（服务可售卖）**
   - 改造点：服务能力从“默认可用”升级为“按套餐/License 授权可用”。
   - 实现方案：
     - 建立产品目录：`product -> plan -> feature -> entitlement`。
     - 每个插件定义能力编码（Feature Code），网关/服务侧做统一鉴权。
     - 提供授权生效链路：购买（或分配）-> 下发授权 -> 实时生效（Redis 缓存 + 事件刷新）。
   - 验收标准：未授权租户访问对应插件接口返回标准化“未开通”错误码。

4. **API 网关统一入口与策略控制**
   - 改造点：所有插件服务纳入统一网关，统一做鉴权、限流、审计。
   - 实现方案：
     - 网关过滤器增加：租户识别、插件授权校验、接口级限流（按租户+接口）。
     - 输出统一审计日志（请求方、租户、插件、耗时、结果）。
   - 验收标准：不经网关无法访问核心 API；审计日志可追溯到租户与用户。

#### P1：服务插件化与平台化能力（本阶段核心）

5. **服务模块插件化（解耦部署与启停）**
   - 改造点：设备管理、数据采集、远程升级、大屏服务从“业务模块”重构为“标准插件”。
   - 实现方案：
     - 定义插件元数据模型：`plugin_code`, `version`, `dependencies`, `api_scopes`。
     - 约定插件生命周期：安装、启用、禁用、升级、卸载（逻辑卸载）。
     - 前端菜单与路由由插件清单动态渲染，避免硬编码菜单。
   - 验收标准：新租户开通插件后，无需改代码即可显示菜单并访问能力。

6. **配置与能力中心（租户级参数）**
   - 改造点：插件参数从“全局配置”升级为“租户级 + 环境级配置”。
   - 实现方案：
     - 配置模型：`tenant_id + plugin_code + key`。
     - 配置缓存（Redis）+ 配置变更事件广播，支持秒级生效。
   - 验收标准：不同租户同一插件可使用不同规则/阈值/回调地址。

7. **计费与用量采集（为商业化做准备）**
   - 改造点：记录租户在每个插件的资源用量。
   - 实现方案：
     - 定义计量维度：设备数、在线时长、采集点位数、OTA 包分发量、报表调用次数等。
     - 埋点上报到用量仓库（可先 MySQL，后续演进至 ClickHouse）。
     - 套餐配额校验（软阈值告警 + 硬阈值拦截）。
   - 验收标准：可按租户、插件、时间维度导出账单明细。

8. **统一任务与事件总线（异步化）**
   - 改造点：远程升级、批量采集、报表计算等重任务异步化。
   - 实现方案：
     - 引入消息队列（Kafka/RabbitMQ 二选一）承载领域事件。
     - 任务状态机标准化：`PENDING/RUNNING/SUCCESS/FAILED/CANCELED`。
     - 幂等机制：任务去重键 + 重试策略 + 死信队列。
   - 验收标准：长耗时任务不阻塞 API，请求超时率显著下降。

#### P2：稳定性与企业级能力（中期）

9. **可观测性与租户级运维**
   - 改造点：监控从“系统级”升级为“租户 + 插件 + 接口”多维。
   - 实现方案：
     - 指标：QPS、错误率、P95 时延、授权失败率、配额触发率。
     - 日志字段统一：`trace_id`, `tenant_id`, `plugin_code`, `user_id`。
     - 告警路由到租户管理员/平台运维。

10. **数据安全与合规能力**
    - 改造点：补齐 SaaS 场景下的数据保护能力。
    - 实现方案：
      - 敏感字段脱敏展示、关键操作二次确认、审计不可篡改。
      - 租户数据导出/删除流程（满足合规与客户退租场景）。
      - 密钥分级管理（应用密钥、设备密钥、OTA 签名密钥）。

11. **高可用与弹性扩缩容**
    - 改造点：面向多租户高并发场景保障 SLA。
    - 实现方案：
      - 服务无状态化 + Redis/DB/MQ 高可用部署。
      - 插件服务按负载独立扩容（如采集与 OTA 服务拆分扩容）。
      - 灰度发布 + 回滚策略（按租户灰度）。

### 三、四大服务插件的落地方案

1. **设备管理插件**
   - 能力范围：设备接入、分组、状态监控、远程控制、工单联动。
   - 关键改造：设备实体绑定 `tenant_id`；设备配额（设备数上限）校验。

2. **数据采集插件**
   - 能力范围：MQTT/HTTP/OPC-UA 数据接入、规则清洗、时序入库、告警。
   - 关键改造：
     - 采集点与规则模板租户隔离。
     - 高吞吐数据链路与冷热分层存储。

3. **远程升级插件（OTA）**
   - 能力范围：升级包管理、灰度发布、批次升级、回滚、结果追踪。
   - 关键改造：
     - 升级包存储与签名校验。
     - 升级任务状态机与失败重试策略。

4. **数据大屏插件**
   - 能力范围：租户自定义看板、模板市场、实时指标、多端展示。
   - 关键改造：
     - 图表数据源权限控制（仅可访问租户可见数据集）。
     - 大屏模板按插件授权与套餐等级开放。

### 四、推荐实施路径（3 个阶段）

- **阶段 1（4~6 周）**：完成 P0
  - 交付：多租户隔离、统一认证授权、插件授权校验、网关统一入口。
- **阶段 2（6~10 周）**：完成 P1
  - 交付：四大服务插件化、租户配置中心、用量计费、任务事件总线。
- **阶段 3（持续演进）**：完成 P2
  - 交付：可观测性、合规安全、高可用与灰度体系。

### 五、关键数据模型建议（最小集合）

- `saas_tenant`：租户主表
- `saas_user_tenant_rel`：用户-租户关系
- `saas_plan`：套餐定义
- `saas_feature`：能力点定义
- `saas_entitlement`：租户已授权能力
- `saas_usage_bill`：用量汇总/账单
- `saas_plugin_registry`：插件注册与版本
- `saas_audit_log`：租户级审计日志

### 六、风险与规避建议

- **风险 1：历史数据无租户字段，改造影响面大**
  - 规避：分批迁移 + 双写过渡 + 回滚脚本。
- **风险 2：插件授权分散在各服务，容易漏校验**
  - 规避：优先在网关做统一校验，服务侧做兜底。
- **风险 3：远程升级任务失败不可控**
  - 规避：任务状态机 + 分批灰度 + 自动回滚策略。
- **风险 4：大屏查询拖垮在线事务库**
  - 规避：读写分离 + 分析型存储 + 缓存预聚合。
